Section 01 · Dart

Dart Language Essentials:
Everything You Need to Read Flutter Code

Flutter is written in Dart. You don't need to master Dart before starting, but you need to recognise its syntax. These are the exact features you'll encounter in every Flutter tutorial and codebase.

Why Dart first?

Every code sample in this guide is Dart. Variables with ?, functions with async, classes with extends — if these look foreign, the Flutter code will be incomprehensible. Spend 30 minutes here and the rest of the guide becomes readable immediately.

Variables & Type System

Dart — Variables, types, var, final, const
// ── Type declarations ─────────────────────────────────────
String  name    = 'Kasun';
int     age     = 25;
double  height  = 1.75;
bool    isAdmin = true;

// var → Dart infers the type (still strongly typed)
var city = 'Colombo';   // inferred as String

// final → set once, cannot change (value decided at runtime)
final uid = 'abc123';   // can't reassign uid after this

// const → compile-time constant (known before app runs)
const appName = 'NoticeBoard';

// ── Null Safety ── the most important Dart concept for Flutter
String  required  = 'always has a value';
String? nullable  = null; // ? means "this CAN be null"

// Accessing nullable: check before use
if (nullable != null) {
  print(nullable.length);    // safe — Dart knows it's not null here
}

// Null-aware operators
print(nullable?.length);      // ?. → only call .length if not null, else null
print(nullable ?? 'default'); // ?? → use 'default' if nullable is null
String safe = nullable!;       // ! → "I KNOW it's not null" (crashes if wrong)

// Late variables: declare now, assign before first use
late String greeting; // not null, but assigned later
greeting = 'Hello, $name!';  // String interpolation: $ inserts a variable

Classes & Constructors

Dart — Classes, constructors, extends, getters
class User {
  final String uid;
  final String name;
  String role;

  // Named constructor using 'this.' shorthand (sets fields directly)
  User({required this.uid, required this.name, this.role = 'user'});

  // Factory constructor: converts a Map (from Firestore) into a User
  factory User.fromJson(Map<String, dynamic> json) => User(
    uid:  json['uid']  as String,
    name: json['name'] as String,
    role: json['role'] as String? ?? 'user',
  );

  // Convert back to Map (for saving to Firestore)
  Map<String, dynamic> toJson() => {'uid': uid, 'name': name, 'role': role};

  // Getter: computed property
  bool get isAdmin => role == 'admin';
}

// Inheritance
class AdminUser extends User {
  AdminUser({required String uid, required String name})
      : super(uid: uid, name: name, role: 'admin');
}

// Usage
final user = User(uid: 'x1', name: 'Kasun');
print(user.isAdmin); // false
final fromMap = User.fromJson({'uid':'x1','name':'Kasun','role':'admin'});

async / await & Futures

Dart — async, await, Future, try/catch, Stream
// A Future is a value that will arrive later (like a promise)
// async marks a function as asynchronous
// await pauses until the Future completes
Future<String> fetchUser(String uid) async {
  await Future.delayed(const Duration(seconds: 1)); // simulate network
  return 'Kasun Perera';
}

// Calling an async function — must also be async, or use .then()
Future<void> loadUser() async {
  try {
    final name = await fetchUser('x1'); // waits here until done
    print('Loaded: $name');
  } catch (e) {
    print('Error: $e');
  }
}

// Stream: a series of Future values over time (like a river)
// Firebase .snapshots() returns a Stream — it keeps emitting updates
Stream<int> counter() async* {
  for (var i = 0; i < 5; i++) {
    await Future.delayed(const Duration(seconds: 1));
    yield i; // emit a value (like return but keeps going)
  }
}

// Listen to a stream
counter().listen((value) => print(value)); // prints 0,1,2,3,4 one per second

// In Flutter, StreamBuilder wraps a Stream in a widget — see §24 (Firestore)
⚡
The two most common Dart patterns in Flutter:
① final x = await someAsyncFunction() — get data once (HTTP, one-time Firestore read).
② stream.listen((data) { ... }) — react to continuous updates (Firebase Auth, Firestore snapshots, GPS).

Section 02 · Dart

Understanding main.dart:
How a Flutter App Boots

main.dart is the very first file every student opens. It contains void main(), runApp(), and MaterialApp — and most beginners have no idea what any of it means. Here is every line, explained.

Dart — Complete main.dart annotated line by line
// Every Dart file that uses Flutter widgets must import this
import 'package:flutter/material.dart';

// ── void main() ───────────────────────────────────────────
// This is where Dart starts running your app.
// It exists in EVERY Dart program. Flutter is no exception.
// void = this function returns nothing
void main() {
  // runApp() takes ONE widget and makes it fill the entire screen.
  // Everything your app shows is a child of this widget.
  runApp(const MyApp());
}

// ── MyApp ─────────────────────────────────────────────────
// A StatelessWidget because the root app itself never changes.
// It just declares the app's configuration.
class MyApp extends StatelessWidget {
  const MyApp({super.key}); // super.key for widget identity — always include

  @override  // signals: we are overriding the parent's build method
  Widget build(BuildContext context) {
    // MaterialApp is the root of a Material Design app.
    // It provides: theme, routing, dark mode, localization,
    // and the Navigator that all screens use to navigate.
    return MaterialApp(
      title: 'My App',                   // shown in task switcher (Android)
      debugShowCheckedModeBanner: false, // hides the red DEBUG banner

      // Theme: Material 3 with a generated colour scheme
      theme: ThemeData(
        useMaterial3: true,
        colorScheme: ColorScheme.fromSeed(seedColor: Colors.deepPurple),
      ),

      // home: the screen shown first when the app opens
      home: const HomeScreen(),
    );
  }
}
App boot sequence
void main()← Dart entry point. App starts here.
└─ runApp(MyApp())← Makes MyApp fill the screen
└─ MaterialApp← Provides theme, Navigator, routes
└─ home: HomeScreen()← First screen shown
└─ Scaffold← AppBar + Body + FAB layout
BuildContext — what is context?

BuildContext context appears in every build() method. It is a handle to the widget's position in the widget tree. Flutter uses it to find: the nearest Theme (for colours and fonts), the nearest Navigator (for navigation), the nearest Provider (for state). Think of it as a GPS coordinate in the widget tree — it tells Flutter where your widget is so it can look up the right ancestor.

Dart — With Firebase init (the async main() pattern)
// When Firebase is added, main() becomes async
// WidgetsFlutterBinding.ensureInitialized() MUST come first
// — it starts Flutter's engine before any async work runs
void main() async {
  WidgetsFlutterBinding.ensureInitialized(); // ← always first if main() is async
  await Firebase.initializeApp(
    options: DefaultFirebaseOptions.currentPlatform,
  );
  runApp(const MyApp());
}

// With Provider wrapping the whole app
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp(options: DefaultFirebaseOptions.currentPlatform);
  final userProv = UserProvider()..init();
  runApp(MultiProvider(
    providers: [
      ChangeNotifierProvider.value(value: userProv),
      ChangeNotifierProvider(create: (_) => ThemeProvider()),
    ],
    child: const MyApp(),
  ));
}

🔥

Section 04 · Flutter

The Golden Rule:
Everything is a Widget

In Flutter, the entire UI — from a button to a full screen — is built from widgets. A widget is simply a description of what to display. Think of them as LEGO blocks you snap together.

📐 What is a Widget?

A widget is a class that describes a part of the UI. It's not the actual pixel on screen — it's the instructions for drawing those pixels. Flutter takes your widget tree and efficiently renders it. Text, images, buttons, padding, colors, entire pages — everything is a widget.

Here's how Flutter widgets nest together to form a complete screen — this is called the Widget Tree:

MaterialApp— Root of your app, sets theme & navigation
└─ Scaffold— Standard page layout (appBar + body)
├─ AppBar— Top navigation bar
└─ Column— Stacks children vertically
├─ Text— Displays a string
├─ SizedBox— Adds empty space
└─ FilledButton— Primary action button
🧱 StatelessWidget
A widget that never changes after it's built. Like a painted wall — it looks the same every time. Perfect for static content like labels, icons, or layouts that don't change. Use this when your UI doesn't react to user interaction.
⚡ StatefulWidget
A widget that can change over time. Like a light switch — it can be ON or OFF. When data changes (e.g., a counter increments), this widget rebuilds to show the new state. Use this for interactive UI.
Dart — StatelessWidget 
// A StatelessWidget — displays content that never changes
class WelcomeCard extends StatelessWidget {
  const WelcomeCard({super.key});

  // build() returns what the widget looks like
  @override
  Widget build(BuildContext context) {
    return Card(
      child: Padding(
        padding: const EdgeInsets.all(16.0),
        child: Text(
          'Welcome to Flutter!',
          style: const TextStyle(fontSize: 20),
        ),
      ),
    );
  }
}
🎯
Rule of thumb: Start with StatelessWidget. Only upgrade to StatefulWidget when you need something on the screen to change (e.g., toggle a checkbox, increment a counter, show/hide content).

child vs children — One of the Most Confusing Things for Beginners

The rule, once and for all

child: takes exactly one widget. The widget wraps something single — a Centre wraps one thing, a Padding wraps one thing, a Container holds one thing.

children: takes a List of widgets. The widget arranges multiple things — a Row lines up many, a Column stacks many, a Stack layers many.

If you pass a list to child: you get a compile error. If you pass one widget to children: you get a compile error. Flutter will tell you immediately — it's enforced at compile time, not runtime.

child: — one widget only
Center(
  child: Text('hello'),
)

Padding(
  padding: EdgeInsets.all(8),
  child: Icon(Icons.star),
)

Container(
  color: Colors.blue,
  child: Text('box'),
)

Column(
  children: [
    Text('First'),
    Text('Second'),
    Text('Third'),
  ],

Dart — child vs children in practice 
// ── child: — wraps exactly ONE widget ────────────────────
Center(child: const Text('I am centred'))

Padding(
  padding: const EdgeInsets.all(16),
  child: ElevatedButton(onPressed: () {}, child: const Text('Tap')),
)

Container(
  width: 200, height: 200,
  color: Colors.blue,
  child: const Icon(Icons.star, color: Colors.white, size: 48),
)

// ── children: — takes a List<Widget> ─────────────────────
Column(
  mainAxisAlignment: MainAxisAlignment.center,
  children: [                    // ← square brackets = List
    const Text('Line one'),
    const SizedBox(height: 8),  // spacer IS a widget too
    const Text('Line two'),
    ElevatedButton(onPressed: () {}, child: const Text('OK')),
  ],
)

// ── The nesting — child and children together ─────────────
// A Column (children:) can contain a Center (child:)
// which itself contains a Text (no child needed — Text is a leaf)
Column(
  children: [
    Center(                       // child: — one thing
      child: Container(           // child: — one thing
        padding: const EdgeInsets.all(12),
        child: Row(               // children: — many things
          children: [
            const Icon(Icons.person),
            const SizedBox(width: 8),
            const Text('Kasun Perera'),
          ],
        ),
      ),
    ),
    const Divider(),              // siblings in the Column
    const Text('Member since 2024'),
  ],
)

// ── Common mistake: forgetting [] around children ─────────
// ✗ WRONG — compile error
Column(children: Text('oops'))  // Text is not a List

// ✓ CORRECT
Column(children: [Text('fixed')])  // wrap in []

// ── Common mistake: using child: on a multi-child widget ──
// ✗ WRONG — Column has no 'child' parameter
Column(child: Text('oops'))    // compile error

// ✓ CORRECT
Column(children: [Text('fixed')])

How to always know which to use: Ask — does this widget arrange multiple things side by side or stacked? Then it's children:. Does it wrap, decorate, or constrain a single thing? Then it's child:. When in doubt, type the widget name and let your IDE autocomplete show you — it will only offer the one that exists.

Section 07 · Flutter

State Management:
Making UIs React to Data

State is data that can change over time. When state changes, Flutter rebuilds the affected widgets to reflect the new data. Understanding this is the key to building interactive apps.

🧠 What is State?

State is any data that influences what the UI shows. Examples: Is the checkbox checked? What has the user typed? Which tab is selected? How many items are in the cart? When this data changes, the UI needs to update — that's state management.

Dart — StatefulWidget Counter (the classic example) 
class CounterPage extends StatefulWidget {
  const CounterPage({super.key});

  @override
  State<CounterPage> createState() => _CounterPageState();
}

class _CounterPageState extends State<CounterPage> {
  int _count = 0; // ← This is our "state"

  void _increment() {
    setState(() {
      // Always change state INSIDE setState()
      // This tells Flutter: "please rebuild the UI!"
      _count++;
    });
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: Center(
        child: Text(
          'Count: $_count', // Displays current state
          style: Theme.of(context).textTheme.displayMedium,
        ),
      ),
      floatingActionButton: FloatingActionButton(
        onPressed: _increment, // calls our method
        child: const Icon(Icons.add),
      ),
    );
  }
}
🔑
The Golden Rule: Never modify state variables directly (_count++ alone). Always wrap the change inside setState(() { ... }). That's what signals Flutter to redraw the widget.

Basic Navigation:
The Navigator Stack

Navigator manages a stack of Route objects. Every screen you push goes on top. Popping removes the top. The user always sees the topmost route. Understanding this stack model is the foundation for everything else.

📚 The Stack Mental Model

Think of a stack of cards. You always see the top card. push places a new card on top. pop removes the top card to reveal the one beneath. pushReplacement swaps the top card. pushAndRemoveUntil clears the deck and starts fresh. The card beneath is always preserved — it remembers its state.

Navigator.push()
Add a new screen on top of the stack. User can go back with the back button. State of the previous screen is preserved.
Navigator.pop()
Remove the top screen. Returns to the one below. Can optionally return a value: Navigator.pop(context, result).
pushReplacement()
Replace the current screen with a new one. No back button to the replaced screen. Used for post-login navigation.
pushAndRemoveUntil()
Push a new screen and remove all screens below it matching a condition. Use (route) => false to clear the entire stack. Used after login to prevent going back to auth screens.
Dart — All Navigator patterns with return values 
// ── 1. Basic push ─────────────────────────────────────────
Navigator.push(context,
  MaterialPageRoute(builder: (_) => const DetailScreen()),
);

// ── 2. Push and wait for a return value ───────────────────
final String? result = await Navigator.push<String>(context,
  MaterialPageRoute(builder: (_) => const PickColourScreen()),
);
if (result != null) print('User picked: $result');

// ── 3. Pop and return a value ──────────────────────────────
Navigator.pop(context, 'red'); // received by the awaiting push above

// ── 4. Replace current screen (no back button) ────────────
Navigator.pushReplacement(context,
  MaterialPageRoute(builder: (_) => const HomeScreen()),
);

// ── 5. Clear stack and push (post-login pattern) ──────────
Navigator.pushAndRemoveUntil(context,
  MaterialPageRoute(builder: (_) => const HomeScreen()),
  (route) => false, // remove ALL existing routes
);

// ── 6. Check if we can pop (is there a screen below?) ─────
if (Navigator.canPop(context)) {
  Navigator.pop(context);
} else {
  // We're at the root — can't go further back
}

// ── 7. Pop multiple screens at once ───────────────────────
Navigator.popUntil(context, ModalRoute.withName('/'));
// pops all screens until the named route '/' is on top

Named Routes:
Centralised Routing

Instead of creating MaterialPageRoute everywhere, define all routes in one place with string names. Navigate using those names from anywhere in the app — cleaner, more maintainable, and required for deep linking.

Dart — Defining and using named routes 
// ── Step 1: Define route constants (avoid typos) ──────────
class AppRoutes {
  static const String home     = '/';
  static const String login    = '/login';
  static const String profile  = '/profile';
  static const String settings = '/settings';
  static const String detail   = '/detail';
}

// ── Step 2: Register all routes in MaterialApp ────────────
MaterialApp(
  initialRoute: AppRoutes.home,
  routes: {
    AppRoutes.home:     (ctx) => const HomeScreen(),
    AppRoutes.login:    (ctx) => const LoginScreen(),
    AppRoutes.profile:  (ctx) => const ProfileScreen(),
    AppRoutes.settings: (ctx) => const SettingsScreen(),
    // Note: routes with dynamic parameters use onGenerateRoute instead
  },
)

// ── Step 3: Navigate using names ──────────────────────────
Navigator.pushNamed(context, AppRoutes.profile);
Navigator.pushReplacementNamed(context, AppRoutes.home);
Navigator.pushNamedAndRemoveUntil(
  context, AppRoutes.home, (route) => false);

// ── Pass arguments to named routes ────────────────────────
Navigator.pushNamed(
  context,
  AppRoutes.detail,
  arguments: {'id': 'abc123', 'title': 'My Item'},
);

// ── Receive arguments in the destination screen ───────────
class DetailScreen extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    final args = ModalRoute.of(context)!.settings.arguments
        as Map<String, dynamic>;
    final id    = args['id']    as String;
    final title = args['title'] as String;
    return Scaffold(appBar: AppBar(title: Text(title)));
  }
}
⚠️ Limitation of static routes map

The routes map cannot handle dynamic parameters like /product/123 — it only works for exact string matches. For dynamic routes, use onGenerateRoute (covered in Section 20). In production apps, consider the go_router package which handles both cases elegantly.

Dynamic Routing:
onGenerateRoute & go_router

When routes depend on runtime data — like /product/42 or /user/kasun — the static routes map isn't enough. onGenerateRoute gives you a function to build routes dynamically. go_router takes this further with URL-based, type-safe routing.

Dart — onGenerateRoute: routing with dynamic segments 
MaterialApp(
  // onGenerateRoute receives a RouteSettings with name + arguments
  // It's called for any route NOT in the routes: map
  onGenerateRoute: (settings) {
    final name = settings.name!;

    // Handle /product/123 ← dynamic segment
    if (name.startsWith('/product/')) {
      final productId = name.split('/').last; // extract "123"
      return MaterialPageRoute(
        builder: (_) => ProductScreen(id: productId),
        settings: settings,
      );
    }

    // Handle named routes with type-safe arguments
    switch (name) {
      case AppRoutes.profile:
        final userId = settings.arguments as String;
        return MaterialPageRoute(
          builder: (_) => ProfileScreen(userId: userId));

      case AppRoutes.home:
        return MaterialPageRoute(builder: (_) => const HomeScreen());

      default:
        return MaterialPageRoute(
          builder: (_) => Scaffold(
            body: Center(child: Text('Page not found: $name'))));
    }
  },
  onUnknownRoute: (settings) => MaterialPageRoute(
    builder: (_) => const NotFoundScreen()),
)
Dart — go_router: the modern recommended approach (flutter pub add go_router) 
import 'package:go_router/go_router.dart';

// ── Define the router once ─────────────────────────────────
final _router = GoRouter(
  initialLocation: '/',
  routes: [
    GoRoute(path: '/',       builder: (ctx, _) => const HomeScreen()),
    GoRoute(path: '/login',   builder: (ctx, _) => const LoginScreen()),

    // Dynamic segment: :id is extracted automatically
    GoRoute(
      path: '/product/:id',
      builder: (ctx, state) {
        final id = state.pathParameters['id']!; // type-safe!
        return ProductScreen(id: id);
      },
    ),

    // Nested routes (ShellRoute for bottom nav)
    ShellRoute(
      builder: (ctx, state, child) => AppShell(child: child),
      routes: [
        GoRoute(path: '/home',    builder: (_, __) => const HomeTab()),
        GoRoute(path: '/search',  builder: (_, __) => const SearchTab()),
        GoRoute(path: '/profile', builder: (_, __) => const ProfileTab()),
      ],
    ),
  ],
);

// ── Wire it in MaterialApp.router ─────────────────────────
MaterialApp.router(routerConfig: _router)

// ── Navigate anywhere ──────────────────────────────────────
context.go('/product/42');        // replace stack entry
context.push('/product/42');      // push on stack
context.go('/product/:id', extra: myObject); // pass extra data
context.pop();                      // go back

Nested Navigators:
Navigation Inside a Tab

A nested Navigator creates a separate navigation stack inside a widget — most commonly inside a tab of a BottomNavigationBar. This lets each tab maintain its own navigation history independently.

🧠 When do you need a nested Navigator?

When you have a BottomNavigationBar with 3 tabs and you want Tab 2 to be able to navigate to a detail screen — without affecting the other tabs or removing the bottom bar. A nested Navigator inside Tab 2 handles its own stack while the parent Navigator holds the tabs. This is how every major app (Instagram, YouTube, Gmail) works.

Dart — Nested Navigator inside a tab (NavigationBar stays visible) 
// Each tab gets its own Navigator with its own stack
class MainScreen extends StatefulWidget {
  const MainScreen({super.key});
  @override
  State<MainScreen> createState() => _MainScreenState();
}

class _MainScreenState extends State<MainScreen> {
  int _tab = 0;

  // One GlobalKey per tab — gives us access to each tab's Navigator
  final List<GlobalKey<NavigatorState>> _navKeys = [
    GlobalKey<NavigatorState>(),
    GlobalKey<NavigatorState>(),
    GlobalKey<NavigatorState>(),
  ];

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: IndexedStack(
        index: _tab,
        // IndexedStack keeps all tabs alive and preserves their state
        children: [
          _buildTab(_navKeys[0], const HomeTab()),
          _buildTab(_navKeys[1], const SearchTab()),
          _buildTab(_navKeys[2], const ProfileTab()),
        ],
      ),
      bottomNavigationBar: NavigationBar(
        selectedIndex: _tab,
        onDestinationSelected: (i) => setState(() => _tab = i),
        destinations: const [
          NavigationDestination(icon: Icon(Icons.home),   label: 'Home'),
          NavigationDestination(icon: Icon(Icons.search), label: 'Search'),
          NavigationDestination(icon: Icon(Icons.person), label: 'Profile'),
        ],
      ),
    );
  }

  // Each tab is wrapped in its own Navigator
  Widget _buildTab(GlobalKey<NavigatorState> key, Widget child) {
    return Navigator(
      key: key,
      onGenerateRoute: (settings) => MaterialPageRoute(
        builder: (_) => child,
        settings: settings,
      ),
    );
  }
}

Navigation Keys:
Navigate Without Context

A GlobalKey<NavigatorState> gives you a direct handle to a Navigator, letting you navigate from anywhere — including Provider callbacks, notification handlers, and background services — without needing a BuildContext.

🧠 Why you need this

You can only call Navigator.push(context, ...) from inside a widget's build tree where a BuildContext is available. But what if a push notification arrives and you need to navigate to a specific screen? Or a Provider method finishes a background task and needs to redirect the user? Navigation keys solve this — they expose the Navigator outside the widget tree.

Dart — GlobalKey<NavigatorState>: navigate from anywhere 
// ── Step 1: Create the key (usually in main.dart or a singleton) ──
final navigatorKey = GlobalKey<NavigatorState>();

// ── Step 2: Pass it to MaterialApp ────────────────────────
MaterialApp(
  navigatorKey: navigatorKey,  // ← links the key to the app's root Navigator
  home: const HomeScreen(),
)

// ── Step 3: Navigate without context — from anywhere ──────
// In a Provider method:
void onAuthSuccess() {
  navigatorKey.currentState!.pushReplacementNamed('/');
}

// In an FCM notification handler (outside widget tree):
FirebaseMessaging.onMessageOpenedApp.listen((message) {
  final screen = message.data['screen'];
  if (screen == 'chat') {
    navigatorKey.currentState!.pushNamed('/chat',
        arguments: message.data['chatId']);
  }
});

// In a background service or timer:
void sessionExpired() {
  navigatorKey.currentState!.pushNamedAndRemoveUntil(
    '/login', (route) => false,
  );
}

// ── Check current route name ───────────────────────────────
String? currentRoute;
navigatorKey.currentState!.popUntil((route) {
  currentRoute = route.settings.name;
  return true; // don't actually pop
});

REST APIs & HTTP:
Fetching Live Data

Most apps need to talk to an external server. Flutter provides the http package for simple requests and dio for production features like interceptors, automatic retries, and cancellation tokens.

The HTTP request cycle

Flutter sends an HTTP request → server returns JSON → jsonDecode() converts it to a Dart Map → a model's fromJson() converts it to a typed object → FutureBuilder displays it. That's the complete loop.

pubspec.yaml

Dart — Model class with fromJson/toJson + GET request
import 'dart:convert';
import 'package:http/http.dart' as http;

// ── Model class ───────────────────────────────────────────
class Post {
  final int    id;
  final String title;
  final String body;

  const Post({required this.id, required this.title, required this.body});

  // fromJson: raw JSON Map → typed Dart object
  factory Post.fromJson(Map<String, dynamic> j) =>
      Post(id: j['id'] as int, title: j['title'] as String, body: j['body'] as String);

  // toJson: typed object → Map for POST body
  Map<String, dynamic> toJson() => {'id': id, 'title': title, 'body': body};
}

// ── GET: fetch a list ─────────────────────────────────────
Future<List<Post>> fetchPosts() async {
  final res = await http.get(Uri.parse('https://jsonplaceholder.typicode.com/posts'));
  if (res.statusCode != 200) throw Exception('HTTP ${res.statusCode}');
  final data = jsonDecode(res.body) as List;
  return data.map((e) => Post.fromJson(e as Map<String, dynamic>)).toList();
}

// ── POST: send data ───────────────────────────────────────
Future<Post> createPost(Post post) async {
  final res = await http.post(
    Uri.parse('https://jsonplaceholder.typicode.com/posts'),
    headers: {'Content-Type': 'application/json'},
    body: jsonEncode(post.toJson()),
  );
  return Post.fromJson(jsonDecode(res.body) as Map<String, dynamic>);
}

// ── FutureBuilder — best practice ────────────────────────
// Store the Future in initState, NOT in build() — otherwise
// it re-calls the API on every single widget rebuild!
class _PostsScreenState extends State<PostsScreen> {

  @override

    future: _future,

      return ListView.builder(
        itemCount: snap.data!.length,
        itemBuilder: (_, i) => ListTile(title: Text(snap.data![i].title)),
      );
    },
  );
}

Implicit Animations:
Flutter Handles the In-Between

Implicit animated widgets automatically animate from their old value to their new value whenever a property changes. You change the value inside setState — Flutter figures out the smooth transition. No controller needed.

How implicit animations work

// Use SingleTickerProviderStateMixin to supply vsync
class _MyWidgetState extends State<MyWidget>
    with SingleTickerProviderStateMixin {

  late AnimationController _ctrl;
  late Animation<double>  _rotation;

  @override
  void initState() {
    super.initState();
    // Controller drives 0.0 → 1.0 over the duration
    _ctrl = AnimationController(
      vsync: this,          // syncs to screen refresh rate
      duration: const Duration(seconds: 2),
    )..repeat();            // loop forever

    // Tween maps controller's 0–1 to your value range
    _rotation = Tween<double>(begin: 0, end: 6.2832)  // 0 → 2π radians
        .animate(CurvedAnimation(parent: _ctrl, curve: Curves.linear));
  }

  @override
  void dispose() { _ctrl.dispose(); super.dispose(); } // ALWAYS dispose

  @override
  Widget build(BuildContext context) {
    // AnimatedBuilder rebuilds only THIS subtree on each frame
    return AnimatedBuilder(
      animation: _ctrl,
      builder: (ctx, child) => Transform.rotate(
        angle: _rotation.value,
        child: child,
      ),
      child: const Icon(Icons.settings, size: 60), // built ONCE — not every frame
    );
  }
}

Section 19 · Provider

Provider: Sharing State
Across Your Whole App

Provider is Flutter's most popular state management solution for beginners and professionals alike. It solves one specific problem: sharing data between widgets that are not directly connected — without messy, fragile workarounds.

The Problem Provider Solves

😤 The "prop drilling" problem

Imagine your logged-in user's name needs to display on a ProfileScreen, an AppBar, and a SettingsScreen. With only setState, you'd have to pass that user object down through every widget in between — even widgets that don't use it at all. This is called "prop drilling" and it makes code messy and hard to maintain very quickly.

❌ Without Provider
MyApp(user)
 → HomeScreen(user)
  → NavBar(user)
   → Avatar(user) ← needs it
Passes through 3 widgets just to reach the one that needs it.
✅ With Provider
ChangeNotifierProvider
 → MyApp
  → HomeScreen
   → Avatar → context.watch()
Avatar reads directly from the Provider. No drilling.

The 3 Concepts You Must Know

📦
ChangeNotifier — the data holder
A class that holds your app's shared data and methods. When data changes, it calls notifyListeners() — which tells all widgets watching it to rebuild. Think of it as the single source of truth.
🏗️
ChangeNotifierProvider — the broadcaster
Wraps part of your widget tree and makes the ChangeNotifier available to every widget inside it. Placed high up in the tree — usually wrapping MaterialApp — so all screens can access it.
👁️
context.watch / context.read — the consumers
context.watch() subscribes to changes — the widget rebuilds when data updates. context.read() accesses the data or calls a method once without subscribing. Use watch for display, read for button callbacks.

Step 2 — Create your ChangeNotifier class

📐 The pattern

Create a class that extends ChangeNotifier. Put your shared data as private fields inside it. Create methods that modify those fields and call notifyListeners() at the end of each method. That's a complete Provider class.

Dart — counter_provider.dart (the simplest possible example) 
import 'package:flutter/foundation.dart';

class CounterProvider extends ChangeNotifier {

  // ── Private data ────────────────────────────────
  int _count = 0;

  // ── Public getter — widgets read this ───────────
  int get count => _count;

  // ── Methods — widgets call these ─────────────────
  void increment() {
    _count++;
    notifyListeners(); // ← tells all watching widgets to rebuild
  }

  void decrement() {
    if (_count > 0) _count--;
    notifyListeners();
  }

  void reset() {
    _count = 0;
    notifyListeners();
  }
}

📌
The golden rule:
• Use context.watch<T>() when your widget needs to display data — it rebuilds when data changes.
• Use context.read<T>() inside button onPressed callbacks — one-time access, no rebuild subscription.

Real-world pattern: User Provider + Firebase Auth

🔥 Provider + Firebase together

The most common real-world use of Provider is storing the currently logged-in user's data so every screen can access it — without re-fetching from Firebase on every screen. The Provider listens to Firebase Auth changes and keeps a UserModel in memory for the whole app.

Dart — user_provider.dart: Firebase Auth + Firestore + Provider together 
class UserProvider extends ChangeNotifier {

  UserModel? _user;        // null when logged out
  bool _isLoading = false;

  UserModel? get user      => _user;
  bool        get isLoading => _isLoading;
  bool        get isLoggedIn => _user != null;
  String?     get role      => _user?.role; // 'victim' | 'volunteer' | 'admin'

  // Call this once in main() — listens to Firebase Auth forever
  void listenToAuthChanges() {
    FirebaseAuth.instance.authStateChanges().listen((firebaseUser) async {
      if (firebaseUser == null) {
        // Logged out — clear user data
        _user = null;
        notifyListeners();
        return;
      }

      // Logged in — fetch full profile from Firestore
      _isLoading = true;
      notifyListeners();

      final doc = await FirebaseFirestore.instance
          .collection('users')
          .doc(firebaseUser.uid)
          .get();

      _user = UserModel.fromFirestore(doc); // your data model
      _isLoading = false;
      notifyListeners(); // every watching widget rebuilds with the new user
    });
  }

  Future<void> signOut() async {
    await FirebaseAuth.instance.signOut();
    // authStateChanges() fires → _user set to null → notifyListeners() called
  }
}

// In main.dart:
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp(options: DefaultFirebaseOptions.currentPlatform);

  final userProvider = UserProvider()
    ..listenToAuthChanges(); // start listening immediately

  runApp(
    ChangeNotifierProvider.value(
      value: userProvider,
      child: const MyApp(),
    ),
  );
}

The Data Model

🧠
Why start here? A model is just a Dart class that represents one piece of data — like a "Notice" object. Defining it first means every other file can use it. The fromFirestore() method converts a raw Firestore document into a clean Dart object. The toMap() method converts it back for saving.
Dart — models/notice_model.dart 
import 'package:cloud_firestore/cloud_firestore.dart';

class NoticeModel {
  final String id;           // Firestore document ID
  final String title;
  final String body;
  final String authorId;     // UID of the user who posted this
  final String authorName;   // Stored so we don't need a 2nd Firestore lookup
  final String? photoUrl;    // nullable — notices don't have to have a photo
  final DateTime? createdAt;

  const NoticeModel({
    required this.id,
    required this.title,
    required this.body,
    required this.authorId,
    required this.authorName,
    this.photoUrl,
    this.createdAt,
  });

  // Convert a Firestore document → NoticeModel Dart object
  // Used when reading from Firestore
  factory NoticeModel.fromFirestore(DocumentSnapshot doc) {
    final data = doc.data()! as Map<String, dynamic>;
    return NoticeModel(
      id:         doc.id,
      title:      data['title']      ?? '',
      body:       data['body']       ?? '',
      authorId:   data['authorId']   ?? '',
      authorName: data['authorName'] ?? 'Unknown',
      photoUrl:   data['photoUrl'],  // null is fine
      createdAt: (data['createdAt'] as Timestamp?)?.toDate(),
    );
  }

  // Convert a NoticeModel → Map for saving to Firestore
  Map<String, dynamic> toMap() => {
    'title':      title,
    'body':       body,
    'authorId':   authorId,
    'authorName': authorName,
    if (photoUrl != null) 'photoUrl': photoUrl, // only include if not null
    'createdAt':  FieldValue.serverTimestamp(), // let Firebase set the time
  };
}

  bool _isLoading = false;

  // Public getters — screens read these
  AppUser? get user      => _user;
  bool      get isLoading => _isLoading;
  bool      get isLoggedIn => _user != null;

  // ── Called ONCE in main.dart — listens to auth forever ──────────────
  // Think of this as: "whenever someone logs in or out, update _user"
  void init() {
    FirebaseAuth.instance.authStateChanges().listen((firebaseUser) async {

      if (firebaseUser == null) {
        // User logged out — clear everything
        _user = null;
        notifyListeners();
        return;
      }

      // User just logged in — fetch their name from Firestore
      _isLoading = true;
      notifyListeners(); // screens can show a loading spinner while waiting

      final doc = await FirebaseFirestore.instance
          .collection('users')
          .doc(firebaseUser.uid)
          .get();

      _user = AppUser(
        uid:   firebaseUser.uid,
        name:  doc['name']  ?? 'User',
        email: doc['email'] ?? firebaseUser.email ?? '',
      );
      _isLoading = false;
      notifyListeners(); // all watching widgets rebuild with the real user data
    });
  }

  // ── Sign Up ──────────────────────────────────────────────────────────
  // Creates a Firebase Auth account AND a Firestore user document
  Future<String?> signUp(String name, String email, String password) async {
    try {
      final cred = await FirebaseAuth.instance
          .createUserWithEmailAndPassword(email: email, password: password);

      // Save their name to Firestore — Firebase Auth doesn't store names
      await FirebaseFirestore.instance
          .collection('users')
          .doc(cred.user!.uid)
          .set({'name': name, 'email': email});

      return null; // null means success — no error message
    } on FirebaseAuthException catch (e) {
      return e.message; // return the error string to show in the UI
    }
  }

  // ── Sign In ──────────────────────────────────────────────────────────
  Future<String?> signIn(String email, String password) async {
    try {
      await FirebaseAuth.instance
          .signInWithEmailAndPassword(email: email, password: password);
      return null; // success
    } on FirebaseAuthException catch (e) {
      return e.message;
    }
  }

  // ── Sign Out ─────────────────────────────────────────────────────────
  Future<void> signOut() => FirebaseAuth.instance.signOut();
  // authStateChanges() fires automatically → _user = null → notifyListeners()
}

Firestore Security Rules for NoticeBoard

🔒
Don't skip this step. Without proper rules, anyone on the internet can read or delete all your data. Paste these rules into Firebase Console → Firestore → Rules and click Publish. These rules enforce: anyone logged in can read notices, only the author can delete their own notice.
Firebase Console → Firestore → Rules 
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Users collection — users can only read/write their own document
    match /users/{userId} {
      allow read, write: if request.auth != null
                         && request.auth.uid == userId;
    }

    // Notices — any logged-in user can read and create
    // but only the author can delete their own notice
    match /notices/{noticeId} {
      allow read, create: if request.auth != null;
      allow delete: if request.auth != null
                    && request.auth.uid == resource.data.authorId;
      allow update: if false; // nobody can edit a notice (by design)
    }
  }

Firebase Authentication:
Login & Signup in Minutes

Firebase Auth handles user accounts completely. You never store passwords — Firebase does it securely. Three concepts to learn: sign up, sign in, and listening to who is currently logged in.

🧠 How Firebase Auth works

When a user signs up, Firebase creates a secure account and gives them a unique UID (User ID) — a string like "abc123xyz". Every piece of data you store for that user is linked to their UID. When they log in later, Firebase verifies their credentials and gives you back that same UID.

Enable Auth in the Firebase Console first

→
Firebase Console → Your project → Authentication → Get started → Sign-in method → Enable Email/Password → Save. That's it — now your app can register users.
Dart — Sign Up, Sign In, Sign Out 
final _auth = FirebaseAuth.instance;

// ── SIGN UP (create new account) ──────────────────
Future<void> signUp(String email, String password) async {
  try {
    final credential = await _auth.createUserWithEmailAndPassword(
      email: email,
      password: password,
    );
    final uid = credential.user!.uid; // e.g. "gH7kLm9..."
    print('New user created! UID: $uid');
  } on FirebaseAuthException catch (e) {
    print('Error: ${e.message}');
    // e.code can be: 'email-already-in-use', 'weak-password'
  }
}

// ── SIGN IN (existing account) ────────────────────
Future<void> signIn(String email, String password) async {
  try {
    await _auth.signInWithEmailAndPassword(
      email: email,
      password: password,
    );
    print('Logged in!');
  } on FirebaseAuthException catch (e) {
    // e.code can be: 'user-not-found', 'wrong-password'
    print('Error: ${e.message}');
  }
}

// ── SIGN OUT ──────────────────────────────────────
Future<void> signOut() async {
  await _auth.signOut();
}

// ── WHO IS LOGGED IN RIGHT NOW? ───────────────────
User? currentUser = _auth.currentUser; // null if not logged in

Dart — Auth State Stream (best practice for routing) 
// authStateChanges() is a Stream that emits:
//   • a User object  → when someone logs in
//   • null           → when someone logs out
// Wrap your root app in this to auto-route between login and home screens

StreamBuilder<User?>(
  stream: FirebaseAuth.instance.authStateChanges(),
  builder: (context, snapshot) {

    // Still loading
    if (snapshot.connectionState == ConnectionState.waiting) {
      return const CircularProgressIndicator();
    }

    // No user → show login screen
    if (!snapshot.hasData) {
      return const LoginScreen();
    }

    // User is logged in → show home screen
    return const HomeScreen();
  },
)

// This is the BEST way to handle login state in Flutter + Firebase.
// When the user logs out, Flutter automatically shows LoginScreen.
// When they log in, it automatically shows HomeScreen.
// And it persists across app restarts — no extra code needed!

Cloud Firestore:
Your App's Database

Firestore is a real-time NoSQL database that stores data as documents inside collections — think of collections as folders and documents as files. Every change syncs to all connected devices instantly.

📂 Collections & Documents — the core idea

A Collection is like a table (e.g., users, posts). A Document is one row in that table (e.g., one specific user). Each document has a unique ID and contains fields (key-value pairs). Documents can also contain sub-collections.

Firestore Structure

├── abc123/ (document — user's UID is the ID)
│ name: "Kasun Perera"
│ email: "[email protected]"
│ role: "volunteer"
└── def456/ (another document)
name: "Amali Silva"
role: "victim"
posts/ (another collection)
└── post001/
title: "Need help!"
authorId: "abc123"

final db = FirebaseFirestore.instance;

  'name': 'Kasun Perera',
  'email': '[email protected]',
  'role': 'volunteer',

Firebase Storage:
Uploading Files & Images

Firebase Storage stores files — photos, PDFs, videos — in the cloud. Your app uploads a file, gets back a URL, and stores that URL in Firestore. Then anyone with the URL can display or download the file.

🖼️ The storage pattern

Upload file → Firebase Storage gives you a download URL → Save that URL as a string in Firestore. To display an image later: read the URL from Firestore → pass it to Image.network(url). Simple.

Dart — Pick an image and upload to Firebase Storage 
import 'package:image_picker/image_picker.dart';
import 'package:firebase_storage/firebase_storage.dart';

Future<String?> uploadProfilePhoto(String userId) async {
  // Step 1: Let user pick a photo from their gallery
  final picker = ImagePicker();
  final picked = await picker.pickImage(
    source: ImageSource.gallery,
    imageQuality: 70, // compress to 70% to save storage
  );
  if (picked == null) return null; // user cancelled

  // Step 2: Create a reference path in Firebase Storage
  // Like a file path: "profile_photos/abc123.jpg"
  final ref = FirebaseStorage.instance
      .ref('profile_photos/$userId.jpg');

  // Step 3: Upload the file
  await ref.putFile(File(picked.path));

  // Step 4: Get the public download URL
  final downloadUrl = await ref.getDownloadURL();
  // e.g. "https://firebasestorage.googleapis.com/v0/b/..."

  // Step 5: Save the URL to Firestore (so you can display it later)
  await FirebaseFirestore.instance
      .collection('users')
      .doc(userId)
      .update({'photoUrl': downloadUrl});

  return downloadUrl;
}

// Display it anywhere with:
Image.network(photoUrl, width: 100, height: 100, fit: BoxFit.cover)

// Or as a CircleAvatar:
CircleAvatar(
  radius: 50,

Firebase Cloud Messaging:
Push Notifications

FCM lets you send notifications to users' phones — even when your app is closed. Free, works on Android and iOS, and takes about 30 minutes to set up.

🔔 How FCM works

Each device gets a unique FCM Token (a long string). Your app saves that token to Firestore. When you want to notify a user, you (or a Cloud Function) sends a message to their token. Firebase delivers it to the device — even if the app is closed.

Dart — FCM Setup: get token + handle messages 
import 'package:firebase_messaging/firebase_messaging.dart';

// ── 1. Request permission (iOS requires this, Android 13+) ──
await FirebaseMessaging.instance.requestPermission(
  alert: true,
  sound: true,
  badge: true,
);

// ── 2. Get this device's FCM token ────────────────
final token = await FirebaseMessaging.instance.getToken();
print('FCM Token: $token');

// Save the token to Firestore so the server knows where to send notifs
await FirebaseFirestore.instance.collection('users').doc(uid).update({
  'fcmToken': token,
});

// ── 3. Handle notification received when app is OPEN ──
FirebaseMessaging.onMessage.listen((message) {
  final title = message.notification?.title ?? 'New alert';
  final body  = message.notification?.body  ?? '';
  // Show a SnackBar or dialog
  ScaffoldMessenger.of(context).showSnackBar(
    SnackBar(content: Text('$title: $body')),
  );
});

// ── 4. Handle notification TAP when app is in background ──
FirebaseMessaging.onMessageOpenedApp.listen((message) {
  // User tapped the notification — navigate to relevant screen
  final screen = message.data['screen']; // data you included
  if (screen == 'task') {
    Navigator.pushNamed(context, '/task-detail');
  }
});

Category 1
Layout Widgets — Structure the UI
Container child:

Box with styling — add color, padding, margin, border, borderRadius, and boxShadow all in one. The Swiss army knife of Flutter layout.

Column children:

Stacks widgets top to bottom. Use mainAxisAlignment to space them (start / center / spaceBetween / spaceEvenly). Most common layout widget in Flutter.

Row children:

Arranges widgets left to right. Same alignment properties as Column but on a horizontal axis. Use for icon + label, avatar + name, price row etc.

Stack children:

Layers widgets on top of each other (Z-axis). Use with Positioned to place items at exact coordinates — text over image, badge on avatar, overlay buttons.

Expanded child:

Fills ALL remaining space inside a Row or Column. Use it to make a TextField fill the width while a button sits beside it. Only works as a direct child of Row/Column.

Flexible child:

Like Expanded but can shrink if the content is smaller than available space. Use flex: to set proportion ratios. Expanded = Flexible(fit: FlexFit.tight).

Wrap children:

Like a Row that wraps onto the next line when it runs out of space. Perfect for tag chips, category pills, badge lists. Use spacing and runSpacing for gaps.

SizedBox child: optional

Either a fixed-size box with a child, or used empty as a spacer: SizedBox(height: 16) is the cleanest way to add vertical gap between widgets in a Column.

Align child:

Positions its child at a specific alignment within its parent: Alignment.topLeft, Alignment.center, Alignment.bottomRight, etc. More precise than Center.

Center child:

Centres its child both horizontally and vertically. Shorthand for Align(alignment: Alignment.center). Use for loading spinners, empty-state messages, hero text.

FutureBuilder
Build UI from a one-time async operation (HTTP call, database read). Handles waiting, error, and data states. Important: store the Future in initState(), not in build().
StreamBuilder
Build UI from a continuous stream of data (Firestore snapshots, websockets, sensors). Rebuilds on every new event. The core pattern for all real-time Flutter apps.
LayoutBuilder
Provides the parent's constraints so you can build different layouts based on available width. Essential for responsive UIs — switch between 1-column and 2-column layouts.
OrientationBuilder
Rebuilds when orientation changes between portrait and landscape. Simpler than LayoutBuilder when you only care about orientation.
ValueListenableBuilder
Lightweight alternative to Provider for simple single-value state. Attach a ValueNotifier and this builder rebuilds only when that value changes. No full Provider setup needed.
AnimatedBuilder
Efficiently rebuild only the animated part of your widget tree. Pass an Animation object and the builder fires every animation frame — only the returning widget rebuilds, not the whole tree.

Platform & Adaptive UI:
Look Native on Every Platform

Flutter runs on Android, iOS, web, macOS, Windows, and Linux. Adaptive UI means your app uses the right components and behaviour for each platform — Material on Android/web, Cupertino on iOS — without maintaining two codebases.

🧠 Three levels of adaptation

Level 1 — Automatic: Material 3 widgets automatically use the platform's visual style for some elements (e.g. CircularProgressIndicator.adaptive()). Level 2 — Conditional: Use Platform.isIOS or defaultTargetPlatform to conditionally show Cupertino widgets. Level 3 — Responsive: Use LayoutBuilder to show different layouts for different screen widths (phone vs tablet vs desktop).

Dart — Platform detection, Cupertino widgets, and adaptive components
